# Дополнение к ТЗ: кроссплатформенная архитектура и облачная синхронизация

> Этот документ не изменяет исходное ТЗ («Разработка защищённого локального хранилища данных с каскадным шифрованием»), а дополняет его архитектурными решениями, которые понадобились из-за требования работать не только на Windows, но и на Mac и телефоне, а также хранить не только логины/пароли, но и произвольный текст и небольшие файлы.

## 0. Контекст и исходные развилки

Исходное ТЗ написано специфично под Windows (`VirtualLock`, `SetWindowDisplayAffinity`, ссылки на DPAPI, `pagefile.sys`). Часть этих API не существует или работает иначе на macOS/iOS/Android. Ниже — как это разрешено.

## 1. Платформенная стратегия

**Решение:** Сейчас делаем полноценное Windows-приложение по ТЗ (.NET 8 / C#), но крипто-ядро и формат контейнера проектируем платформонезависимыми, чтобы контейнер и логика в будущем без переделки заработали на Mac и телефоне.

Архитектурное разделение на три слоя:

1. **Core (platform-agnostic)** — вся криптография (AES-256-GCM, XChaCha20-Poly1305, Argon2id, HKDF), формат контейнера, chaffing/winnowing, работа с индексом смещений. Пишется на .NET 8 без обращений к Windows API, поэтому в будущем компилируется и работает и на Mac/Linux/мобильных платформах (.NET MAUI/Avalonia/Xamarin используют тот же рантайм).
2. **Platform Security Adapter** — интерфейс `IPlatformSecurity` с методами вроде `LockMemory` / `UnlockMemory` / `ZeroMemory` / `ProtectWindowFromCapture` / `ClearClipboardAfter`. Под Windows он реализуется через `VirtualLock` / `SecureZeroMemory` / `SetWindowDisplayAffinity(WDA_MONITOR)`, как требует ТЗ. Когда дойдёт очередь до Mac/телефона — под них пишутся отдельные реализации того же интерфейса (`mlock`, `NSWindow.sharingType`, Android `FLAG_SECURE` и т.п.), а Core и UI-логика не меняются.
3. **UI-слой** — сейчас WPF/WinUI на Windows; в будущем — своя UI-оболочка под Mac/телефон поверх того же Core.

Важно: формат самого файла-контейнера должен быть чистым бинарным форматом без Windows-специфичных метаданных (никаких ADS, DPAPI-блобов, путей). Тогда сейф, созданный на Windows, будет читаться будущими клиентами без миграции данных.

**Известное ограничение на будущее:** iOS технически не позволяет полностью заблокировать скриншот окна (в отличие от Windows `WDA_MONITOR`) — можно только детектировать факт скриншота постфактум. Это нужно будет явно принять как ограничение мобильной версии, когда до неё дойдёт очередь, а не сейчас решать.

## 2. Область хранимых данных

Контейнер должен поддерживать не только структурированные записи логин/пароль/URL/теги (п.5-6 исходного ТЗ), но и:
- произвольные текстовые заметки;
- небольшие вложенные файлы (документы, фотографии) в разумном количестве и объёме — контейнер не рассчитан на роль полноценного файлового хранилища.

Это определяет требования к схеме chaffing/padding — см. п.4.

## 3. Синхронизация между устройствами

Приложение не ведёт собственных аккаунтов и не имеет встроенной автосинхронизации. Перенос контейнера между Windows/Mac/телефоном — сознательно ручной/полу-ручной шаг пользователя, с выбором одного из двух способов:

### 3.1. Способ A — WebDAV с паролем приложения Яндекс.Диска
Яндекс.Диск поддерживает WebDAV (`webdav.yandex.ru`) и генерацию отдельного «пароля приложения» в настройках Яндекс ID — без OAuth-флоу внутри нашего приложения и без хранения основного пароля от аккаунта. Приложение делает обычные WebDAV `PUT`/`GET` для заливки и скачивания контейнера напрямую с любого устройства.

- Пароль приложения по умолчанию **не кэшируется** — вводится вручную при каждой операции синхронизации, по той же логике, что и файл-ключ в п.3.2 исходного ТЗ (принцип «знание из головы пользователя не должно оседать на диске»).
- Опционально (но не по умолчанию) можно предусмотреть хранение пароля приложения в системном хранилище учётных данных ОС (Windows Credential Manager / macOS Keychain) — это отдельное решение, которое стоит принять осознанно, т.к. оно ослабляет модель «ничего не хранится на диске».

### 3.2. Способ B — импорт по прямой ссылке
Пользователь вручную загружает файл-контейнер в Яндекс.Диск (через сайт/десктопный клиент/приложение) и получает публичную ссылку «поделиться». В нашем приложении есть только функция **«Импорт по ссылке»** — обычный HTTPS GET по указанному URL и сохранение результата как локального контейнера. Никакой авторизации в приложении не требуется.

**Рекомендация по безопасности:** сам контейнер уже защищён каскадным шифрованием, так что Яндекс (или любой, кто перехватит трафик/ссылку) видит только шум высокой энтропии. Но публичная ссылка сама по себе — это bearer-токен: у кого есть ссылка, тот может скачать зашифрованный блоб. Стоит:
- защищать публичную ссылку паролем (Яндекс.Диск это поддерживает для «Поделиться»);
- не публиковать/не индексировать такие ссылки;
- рассматривать ссылку как «второй по значимости секрет» после мастер-пароля и файла-ключа.

Оба способа (A и B) выбираются пользователем в UI — жёсткой привязки к одному нет.

## 4. Адаптивная схема chaffing/padding вместо фиксированных 10 МБ

Ожидаемое содержимое (текст, заметки, логины/пароли, немного мелких фото и текстовых файлов) обычно укладывается в единицы мегабайт. Жёсткое округление каждого контейнера до 10 МБ (как буквально написано в п.5 ТЗ) даёт либо too much шума на маленьких сейфах, либо тесноту при росте данных. Вместо этого:

- Реальный размер контейнера квантуется вверх до ближайшего значения из заранее заданной **прогрессии бакетов** (аналог схемы PADMÉ / bucket padding, используемой для защиты от анализа объёма данных по размеру файла): например `256 КБ → 512 КБ → 1 МБ → 2 МБ → 4 МБ → 8 МБ → 16 МБ → ...`
- Наблюдатель, видящий файл контейнера, узнаёт только «между N и 2N», а не точный объём данных — требование п.5 ТЗ по маскировке размера выполняется.
- Свободное место внутри текущего бакета всегда забито CSPRNG-мусором (chaffing), как и предусмотрено ТЗ.
- Когда данные перестают помещаться в текущий бакет, контейнер один раз пересобирается (re-chaff) до следующего бакета — рост «шагами», а не постоянное разрастание на фиксированную величину.
- Конкретную прогрессию бакетов и стартовый размер нужно зафиксировать перед началом разработки (вынесено в открытые вопросы ниже).

## 5. Модель восстановления доступа

Подтверждена строгая zero-knowledge модель из исходного ТЗ: **восстановление невозможно** при утере мастер-пароля или файла-ключа. Recovery-коды, резервные копии мастер-ключа и подобные механизмы сознательно не реализуются, так как они являются дополнительным секретом, ослабляющим модель угроз. UI при создании сейфа должен явно и однозначно предупреждать об этом и требовать осознанного подтверждения от пользователя (чекбокс/фраза-подтверждение, а не просто мелкий текст).

## 6. Открытые вопросы к финализации перед стартом разработки

Эти пункты стоит закрыть явным решением до написания кода (не критично для архитектуры «в целом», но влияет на детали реализации):

1. Хранить ли WebDAV-пароль приложения между сессиями (удобство) или требовать ввод каждый раз (строже, в духе п.3.2 ТЗ про файл-ключ)?
2. Один контейнер на пользователя или поддержка нескольких независимых сейфов/профилей в одном приложении?
3. Точная прогрессия бакетов и стартовый размер контейнера (п.4) — предлагается зафиксировать степенную прогрессию с шагом ×2, стартуя от 256 КБ.
4. Формат публичной ссылки для способа B — предусмотреть ли в UI поле «пароль на ссылку», который приложение подсказывает пользователю выставить на стороне Яндекс.Диска.

## 7. Соответствие исходному ТЗ

Все требования разделов 2–6 исходного ТЗ (каскадное AES-256-GCM + XChaCha20-Poly1305 шифрование, Argon2id KDF с файлом-ключом без кэширования пути, in-memory security с `VirtualLock`/`SecureZeroMemory`, структура контейнера с chaffing/winnowing, защита UI от скриншотов и очистка буфера обмена через 15 секунд) остаются в силе без изменений для Windows-реализации. Данный документ добавляет только: (а) слой абстракции для будущей кроссплатформенности, (б) механизм ручной облачной синхронизации без встроенных аккаунтов, (в) уточнённую адаптивную схему padding вместо фиксированных 10 МБ.
